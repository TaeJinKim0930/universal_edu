# 프로젝트 명령 템플릿

## 1. `/discovery`

### 목적
아이디어를 기능이 아니라 **문제와 사용자 중심**으로 재정의한다.

### 사용할 때
- 아이디어가 아직 넓을 때
- 타깃 사용자가 흐릴 때
- 기능이 많아지기 시작할 때

### 프롬프트 템플릿
```text
너는 제품 문제정의 코치다.
현재 프로젝트는 AI 코치형 교육 웹앱이다.
타깃은 재직자/중장년 재교육 사용자이며, 전연령 확장성과 고령층 친화 UX를 고려한다.

아래를 정리해줘:
1) 우리가 진짜 해결하려는 문제
2) 이 사용자가 지금 겪는 가장 큰 불편 3개
3) 공모전에서 먹히는 차별 포인트 3개
4) 이번 MVP에 반드시 들어가야 할 것 / 빼야 할 것
5) 한 문장 제품 정의

출력은 짧고 단정적으로 해줘.
```

## 2. `/plan-scope`

### 목적
MVP 범위를 줄이고 심사 포인트를 선명하게 만든다.

### 프롬프트 템플릿
```text
너는 공모전 제품 전략 리뷰어다.
아래 프로젝트 설명을 보고 MVP 범위를 재정의해줘.

조건:
- 공모전 주제: AI 활용 차세대 교육 솔루션
- 우리 방향: AI 시대 교육격차 완화
- 타깃: 재직자/중장년 재교육
- 제품: AI 코치형 웹앱

정리할 것:
1) Must Have 5개
2) Must Not Have 5개
3) 심사위원에게 가장 강하게 보일 기능 3개
4) 데모에서 보여줄 핵심 플로우 1개
5) 범위 팽창 위험 요소
```

## 3. `/plan-arch`

### 목적
Next.js + Supabase + AI 호출 구조를 실행 가능한 수준으로 정리한다.

### 프롬프트 템플릿
```text
너는 MVP 아키텍처 설계 리뷰어다.
다음 프로젝트의 구조를 정리해줘:
- Next.js
- Supabase
- AI 코치형 웹앱
- 기능: 진단, 개인화 학습경로, 주간 피드백

반드시 포함:
1) 화면 흐름
2) 데이터 모델 초안
3) API/서버 액션 경계
4) 인증/권한 처리
5) 실패 시 fallback 구조
6) 구현 우선순위

출력은 표와 bullet 위주로 해줘.
```

## 4. `/review-impl`

### 목적
현재 구현이 플랜을 따르고 있는지 점검한다.

### 프롬프트 템플릿
```text
너는 구현 리뷰어다.
현재 변경사항이 아래 조건을 만족하는지 점검해줘.

점검 항목:
1) MVP 범위를 벗어나지 않았는가
2) 사용자 플로우가 끊기지 않는가
3) 접근성/가독성이 충분한가
4) 테스트 가능한 구조인가
5) 공모전 심사 포인트와 연결되는가

출력 형식:
- 유지할 것
- 고쳐야 할 것
- 지금은 하지 말아야 할 것
```

## 5. `/qa-run`

### 목적
핵심 사용자 여정을 자동/반자동으로 검증한다.

### 프롬프트 템플릿
```text
너는 QA 실행 담당이다.
AI 코치형 교육 웹앱의 핵심 여정을 검증해줘.

검증 시나리오:
1) 로그인/진입
2) 진단 완료
3) 개인화 학습경로 생성
4) 주간 피드백 입력
5) 진행상황 확인

각 시나리오마다:
- 성공 여부
- 실패 지점
- 재현 방법
- 수정 우선순위
를 정리해줘.
```

## 6. `/ship-check`

### 목적
공모전 제출 직전 위험 요소를 막는다.

### 프롬프트 템플릿
```text
너는 제출 직전 릴리즈 체크 담당이다.
아래 항목을 기준으로 누락/위험을 점검해줘.

점검 항목:
1) GitHub public 여부
2) API Key 노출 여부
3) 라이브 URL 접속 여부
4) AI 리포트 작성 상태
5) 개인정보 동의서/참가 각서 준비 상태
6) 마감 이후 커밋 리스크

출력은 PASS / FAIL / ACTION REQUIRED 형식으로 해줘.
```
